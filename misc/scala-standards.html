Coding Standards

The philosophical approach and coding standards in Slate Kit

Simple and Practical


The high-level design approach of Slate Kit primarily involves an emphasis on simplicity and practicality while following best practices. There are certain items we avoid or specifically use ( see below ). We try to maintain a healthy balance between delivering new features, implementing new concepts, supporting valid and common use-cases, and incorporating functional programming as much as possible. Some of our other design choices include the following:

Approaches

1. Easy to read code ( for code maintainance by anyone)
2. Pragmatic decisions ( consider costs, trade-offs, learning curves, complexity )
3. Modular components ( that can further be composed by end-users )
4. Minimal dependencies ( to avoid a large dependency chain )
5. Refactor code in the beginning of a new release

1. Avoid

To be in-line with functional programming, the following are either avoided completely or minimally used, to prevent side-effects and chaotic flow-control.

return

 Avoid return expressions to prevent early exits or short circuiting of logic.

null

 Avoid use of null and instead use Option[T] and/or our own Result[T].

throw

 Avoid throwing exceptions and instead use functional error-handling ( Option[T] or our own Result[T] )

var

 Avoid vars as much as possible. Vars are kept to a bare minimum, mostly in low-level, high-performance text parsing.

for

 Prefer foreach,map,reduce,fold, and aggregate functions over the typical for loop.

if

 Prefer map/folds/getOrElse over if/else expressions

try

 Prefer scala's Try or our own attempt function that is an alternative to language level try/catch expressions

2. Use

The following items are used and/or preferred.

val

 Enforce immutability as much as possible.

match

 Use pattern matching over if/else and type checking

map

 Use map, reduce, filter where applicable

aggregates

 Use aggregate functions find, filter, indexOf, min, max, minBy, maxBy etc

for { }

 Use for comprehensions for easier chained use of map/flatmap

case

 Use case classes for easier immutable functionality

Future[T]

 Use Futures for asynchronous processing.
