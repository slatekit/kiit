Team

The high-level design approach of Slate Kit primarily involves an emphasis on simplicity and practicality while following best practices. There are certain items we avoid or specifically use ( see below ). We try to maintain a healthy balance between delivering new features, implementing new concepts, supporting valid and common use-cases, and incorporating functional programming as much as possible. Philosophy

1. Easy to read code for easy code maintenance ( no fancy syntax, minimize jargon, reduce noise )
2. Pragmatic decisions factor cost of design, use-cases, learning curves, complexity
3. Modular components reasonably sized and independent components that can be extended
4. Minimal dependencies careful use of 3rd party libraries, to avoid a large dependency chain.
see our setup for more detail.



Standards

Our coding standards emphasize the following:

1. immutability Generally avoiding vars as much as possible in favour of vals
2. functional error-handling Avoid throwing of exceptions in favour of using our Result[T] class that models success/failures.
3. higher-order functions Allowing functions as inputs and return values
4. testability Ensuring components are designed for testability

However, we are very careful to not be dogmatic about FP ( functional programming ), and do not enforce pure FP.


Avoid

The following are either avoided completely or minimally used.

var

 Avoid var as much as possible. Vars are kept to a minimum, mostly in low-level, text parsing.

null !!

 Avoid use of !! access operator and instead use ? or let.

throw

 Avoid throwing exceptions and instead use functional error-handling ( we use our Result[T] class )

for

 Prefer foreach,map,reduce,fold, and aggregate functions over the typical for loop.

try

 Prefer our attempt function or our own attempt function that returns our Result[T] class an alternative try/catch expressions where applicable.



Use

The following items are used and/or preferred.

val

 Enforce immutability as much as possible.

when

 Use pattern matching over if/else and type checking

map

 Use map, reduce, filter, fold where applicable instead of loops

aggregates

 Use aggregate functions find, filter, indexOf, min, max, minBy, maxBy etc

data

 Use data classes for easier immutable functionality

HOF

 Use higher order functions ( functions as parameters/return values )where applicable.

Async

 Use Kotlin Coroutines/JVM Futures for asynchronous processing.

More

The following concepts, while currently used, will be incorporated in more areas in the near future

Type Safety

 Prefer class types rather than Strings/Numbers to represent values.
E.g. Represent an email address via an Email data class vs a String

Type Builders

 Kotlin Builders will be used for constructing various components

Tail Recursion

 Tail recursion as an alternative to other looping constructs and to avoid vars.

IO Abstraction

 Use a function / callback to abstract IO when applicable. This facilitates unit-tests.
E.g. writer:(String) -> Unit instead of println directly or
E.g. writer:(Request) -> Response instead of Http.post directly
