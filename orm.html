Slate Kit

ORM

A light-weight single table object-relational mapper for cloud databases

Overview

The Scala ORM component in Slate Kit is designed with simplicity in mind. Instead of an elaborate ORM, we focus on the basic features to support single-table based mapping of objects to tables or any other underling datasource. The ORM is designed in form of an "Entities" based approach ( as you will see in the docs ). The following features are supported:


Features

Feature About
SetupRegistering 1 or more database to the ORM
-Connecting a model with a specific data-source
-Auto-mapping to mysql tables
-Setting up your field mappings programmaticaly
-Setting up your field mappings via annotations
UsageYou can use just an individual part of the ORM or the whole thing
-The ORM is separated into a Model definition, Mapper, Repository and Service per model
-Typically all you need is a model and your own Service for a model. E.g. User, UserService
-All entities have an "id" field and timestamps/audit fields ( created, updated, createdBy, updatedBy )
-CRUD operations, FindBy field and DeleteBy field methods are supported.
-A "query" specification pattern is used for more complex queries.
-Outside of these built-in features, you are free to use stored procs, sql
-Reference other model services to perform joins

1. Databases

You can register your entities to use 1 database or many by simply registering your databases by name and/or shard. Then you connect a specific model/entities to a named database. This approach allows for flexibility in supporting different databases.

See docs

4. Mapper

The mapper component is used map fields in the Model definition to/from sql. The Mapper is designed to be general enough to support conversion to/from other formats such as JSON. You can write your own mapper. There is a built in Mapper for MySql.

See docs

2. Models

In order to use the ORM to map entities to tables, you have setup a model definition. A model definition is the name of the model along with the fields to be persisted. This can be done 2 different ways either programmatically or via annotations.

See docs

5. Repo

The Repository layer/component support all of the CRUD operations on your entities and uses a Mapper to Map the entities to/from database records. You don't need to implement your own Repo unless for more complex queries or calling stored procs.

See docs

3. Entities

The entities you want to be auto-mapped to a table need to implement the IEntity trait. This ensure for consistent field names such as "id" ( long ) and timestamp/audit field like created, createdby etc. It also allows for the ORM to use generics.

See docs

6. Service

The Service layer/component serves as the location to put any business logic/operations before/after any CRUD operations. There are already several CRUD methods available here that simply delegate to the Repo, but can override the behaviour.

See docs


Relations

While joins/relationships are not supported currently, you can easily get a reference to other entity Services to query related data. Part of the reason for not initially supporting joins/relationships was to get a simple ORM working well for single-table models for mostly Cloud/Mobile Backend data. The ORM will be enhanced in future versions.


Components

These parts of the ORM were designed to be used independently where applicable while also integrated together to support typical ORM features.


Setup

You can setup the ORM / Entities system to use multiple databases and also connect repositories to different types of databases.

Notes

  • The entities ORM is model using a Model, Repository, Service approach.
  • Mapping of fields is done by placing annotations on the model fields
  • You only need to implement the model. You can use base Repository and Service implementions.
  • You can create you own custom Service implementation
