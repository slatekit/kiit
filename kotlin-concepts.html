Key concepts in Slate Kit

Overview

There are just a few key concepts in Slate Kit that you need to know to get up and running quickly.


Concept About More
1. Env Represents a logical environment such as local, dev, qa, prod more
2. Config Configuration settings as a thin layer on Java Properties with extra features. There is 1 config file per environment more
3. Context The AppContext is a container for common dependencies ( env, config, logger, encryptor ) and is passed to components. more
4. Cloud Slate Kit integrates with Cloud infrastructure such as File Storage and Queues. AWS ( S3, SQS ) is the default but its abstracted so you can easily customize to other providers. more
5. Apis APIs in Slate Kit are are normal methods on classes and are "protocol independent" which means they can be made accessible on the CLI ( command line ) or as an HTTP Endpoint more
6. Routes Abstracts a request whether it is on the CLI ( command line ) or a HTTP Request. This is used to build Slate Kit APIs ( which are protocol independent ). more
7. Request Abstracts a request whether it is on the CLI ( command line ) or a HTTP Request. This is used to build Slate Kit APIs ( which are protocol independent ). more
8. Result Abstracts a response and models success/failures, whether it's on the CLI (Command Line) or an HTTP Response. This is used to build Slate Kit APIs and is compatible with HTTP status codes more


1. Envs

You can setup different environments ( local, dev, qa1, qa2, etc ) for your application. This is accomplished using the Env component below. It is used in the Slate Kit App ( Base App )

Features

Typed Environment name and mode is type-safe
Validated Environment can be validated at startup
About Clear description of the environment
Defaults Choose from defaults available
Names Allow for multiple environments by name and mode

Source

package slatekit.common.envs
jar slatekit-common.jar
source slatekit.common.envs

Example

  /**
   * The available environments to choose from.
   * An environment key is built name and mode. e.g. "qa1.QA"
   *
   * Each of these environments should map to an associated env.{name}.conf
   * config file in the /resources/ directory. 
   *
   * e.g.
   * /resources/env.conf     ( common      config )
   * /resources/env.dev.conf ( development config )
   * etc ..
   *
   * @return
   */
  fun defaults(): Envs =
      Envs( listOf(
          Env("loc", Dev , desc = "Dev environment (local)"),
          Env("dev", Dev , desc = "Dev environment (shared)"),
          Env("qa1", Qa  , desc = "QA environment  (current release)"),
          Env("qa2", Qa  , desc = "QA environment  (last release)"),
          Env("stg", Uat , desc = "STG environment (demo)"),
          Env("pro", Prod, desc = "LIVE environment")
      ))

2. Config

All the different types of Apps and Sample Apps using a common configuration scheme. For each app, there is a single common configuration file env.conf and then 1 environment specific configuration file for each environment. e.g. env.dev.conf All the environment configs are located in the resources directory but you can put them and reference them anywhere. Slate Kit configs are simple abstractions over Java properties files. It is used in the Slate Kit App ( Base App ).

Features

Typed Get typed values such as bool, int, long etc.
Nullable Get nullable values such as Int?
Defaults Get default values when unavailable
Lists Get lists of basic types
Maps Get maps of basic types
Decryption Auto decrypt encrypted strings
Objects Auto load objects from config such as Db connection, Api-Keys
URIs Reference files in user directory to load objects for added security

Source

package slatekit.common.conf
jar slatekit-common.jar
source slatekit.common.conf

Example

  // Examples of just a few features...
  // 1. Typed settings
  // 2. Nullable
  // 3. Defaults 
  // 4. Lists/Maps 
  // 5. Encryption/File Refs/Inheritance
  
  // Load the env.dev.conf file from /resources
  val conf = Config("env.dev.conf")

  // Get typed value: non-nullable
  // NOTE: default value for typed returned if unavailable
  println("bool  : " + conf.getBool("bVal"))
  println("string: " + conf.getString("sVal"))
  println("short : " + conf.getShort("hVal"))
  println("int   : " + conf.getInt("iVal"))
  println("long  : " + conf.getLong("lVal"))
  println("float : " + conf.getFloat("fVal"))
  println("double: " + conf.getDouble("dVal"))
  
  // Get typed value: with default value if unavailable.
  println("bool  : " + conf.getBoolOrElse("bVal", false))
  println("string: " + conf.getStringOrElse("sVal", "abc"))
  println("short : " + conf.getShortOrElse("hVal", 0))
  println("int   : " + conf.getIntOrElse("iVal", 0))
  println("long  : " + conf.getLongOrElse("lVal", 0))
  println("float : " + conf.getFloatOrElse("fVal", 0.0f))
  println("double: " + conf.getDoubleOrElse("dVal", 0.0))
  
  // Get lists/maps
  println("list  : " + conf.getList("listValInt", Int::class))
  println("map   : " + conf.getMap("mapValInt", String::class, Int::class))
  println()
  
  // More features such as decryption, file refs and more.
  // ...

3. Context

The context is a container for global objects needed for most components. Such items include the current environment ( dev, qa, pro, etc ), config settings, a logger, database info, encryptor, the host computer, the runtime version of java, and access to the Entities that provide the ORM entity services. It is created at startup and is read-only.

Features

Container Conveniently store app level dependencies.
Immutable Immutable data can be safely passed around
Global Application level can be globally available
Portable Easily pass around a single context vs many dependencies

Source

package slatekit.core.common
jar slatekit-core.jar
source slatekit.core.common

Example

  AppContext(
      arg = Args.default(),
      env = Env("local", Dev),
      cfg = Config(),
      log = LoggerConsole(),
      ent = Entities(),
      inf = About("myapp", "App1", "Sample app", "Company1, "", "", "", "", "", "", ""),
      host = Host.local(),
      lang = Lang.kotlin(),
      enc = Encryptor("wejklhviuxywehjk", "3214maslkdf03292"),
      dirs = Folders.userDir("slatekit", "apps", "sample1")
  )

4. Cloud

Slate Kit integrates with cloud based file storage and message queues. This is implemented via abstraction layers for both files and queues while providing default implementations for AWS.

Features

Abstraction Provides an abstraction layer to files and queues while supporting most common operations and use-cases
Defaults Sensible default implementations provided for AWS S3 and SQS.
Extendible Easily extends the base classes to integrate with other providers such as Google Cloud or Microsoft Azure.
Portable With the abstractions and extensibility, you can easily switch providers while keeping most of your code intact

Source

type package jar source
Abstractions slatekit.core.cloud slatekit-core.jar slatekit.core.cloud
Implementation ( AWS ) slatekit.cloud.aws slatekit-cloud.jar slatekit.cloud.aws

Example


    // Not storing any key/secret in source code for security purposes
    // Setup 1: Use the default aws config file in "{user_dir}/.aws/credentials"
    val files: CloudFiles = AwsCloudFiles("app1-dev-folder-events", false)

    // Use case 1: create using just name and content
    files.create("2016_nba_v3", "version 1")

    // Use case 2: update using just name and content
    files.update("2016_nba_v3", "version 2")

    // Use case 3: get file as a text using just name
    files.getAsText("2016_nba_v3")

    // Use case 4: download file to local folder
    files.download("2016_nba_v3", "c:/dev/temp/")

    // ... more use cases supported

5. APIs

APIs are a core architectural component of Slate Kit. One of the main design goals of builing APIs in Slate Kit is allow any API to be hosted as a Web/HTTP API and/or hosted on the CLI ( Command Line ) for easy accessibility. We call this approach Protocol Independent APIs.

Why make APIs available on the CLI ?
This provides enormous benefits which include:

  1. Accessibility to your APIs ( for testing / debugging )
  2. Support for API discovery and documentation
  3. Securing and segregating of APIs into local, private, public
  4. Convenient CLI access to your entire application

Features

Unified Routes Unified routing ( see below ) to support both HTTP and CLI
Unified Requests Requests are abstracted out using the Request class ( see below ) to allow your APIs to be accessible via any protocol ( currently only HTTP and CLI supported )
Unified Responses Responses are abstracted out using the Result class ( see below ) to allow your APIs to return meaningful successes/failures that can be easily converted to HTTP and CLI responses.
POKO You can easily build APIs using plain old Kotlin objects with simple annotations
Discoverable APIs become easily discoverable ( see routing below )
Auto Mapping The system will auto-convert HTTP parameters( query string ) and JSON data to your API actions/methods.

Source

package slatekit.apis
jar slatekit-apis.jar
source ApiContainer.kt

Example

 
  @Api(area = "app", name = "users", desc = "manage users",
       roles= "admin", auth = "app", verb = "*", protocol = "*")
  class UserApi(ctx:AppContext) : ApiBaseEntity<User>(ctx, User::class) {
  
      @ApiAction(name = "", desc = "activates a users account", roles= "@parent")
      @ApiArg("phone", "the phone number", true, "1-234-567-8901")
      fun activate(phone:String, code:Int, isPremiumUser:Boolean): Result<Boolean> {
          return ok("activated $phone, $code, $isPremiumUser")
      }
    // ...
  }

6. Routes

In Slate Kit, all requests ( HTTP or CLI ) confirm to what is called the "Unified Routing Format" which is a 3 part routing structure defined below. This simple yet convenient approach ( while not fully REST Compliant by design ) is one part that helps make Slate Kit APIs protocol independent and thus available on both HTTP and CLI.

area | api | action

e.g. app/users/activate 

 // NOTES: 
 // - Slate uses a unified routing format ( area/api/action ) 
 //   for both Http and CLI requests.
 // - See the "Unified Routing" section for more info.
 
 // 1. app.users.activate via Http GET ( query string )
 http://mycompany.com/api/app/users/activate?email=kishore@abc.com
 
 // 2. app.users.activate via Http POST ( json data 
 http://mycompany.com/api/app/users/activate { email="kishore@abc.com" } 
 
 // 3. app.users.activate via CLI 
 C:\apps\app1> app.users.activate -email="kishore@abc.com"

More info: Routes

On the CLI you can easily discover the areas, apis, and actions by simply typing a ? next to the part. e.g.

  1. ?
  2. area?
  3. area.api?
  4. area.api.action?



7. Request

The Request class abstracts a request whether it is on the CLI ( command line ) or a HTTP Request. This is used to build Slate Kit APIs ( which are protocol independent ).

Features

API Design One of the fundamental data structures for making APIs "protocol independent"
Uniformity Allows 1 model to be uniformly used to represent both HTTP and CLI requests.
Immutable Immutable data can be safely passed around
Routing Has a built in 3 part routing structure ( area.api.action ) to standardize the routing for both HTTP and CLI.

Source

package slatekit.common
jar slatekit-common.jar
source Request.kt

Example

  // The request can be built to represent requests from different sources
  // 1. http: http://abc.com/api/app/users/activate { post data }
  // 2. cli : app.users.activate -userId=5001 
  val request = Request(
      path   = "app.users.activate",
      parts  = listOf("app", "users", "activate"),
      area   = "app",
      name   = "users",
      action = "activate",
      verb   = "post",
      opts   = InputArgs(mapOf("api-key" to "ABC-123")),
      args   = InputArgs(mapOf("userId" to 5001)),
      raw    = "the raw HTTP SparkJava request or CLI request",
      tag    = Random.stringGuid()
  )

8. Result

The Result represents an abstraction for a response and also models successes and failures using a combined set of fields for a code, value, message, and success/failure flag. It is general purpose enough to be used anywhere. However, in Slate it is used in conjunction with the Request to allow Slate Kit APIs to be "protocol independent", allowing APIs to run as both HTTP Endpoints and on the CLI (Command Line Interface)

Features

Success/Failure Can represent many types of successes and failures
Modeling Functions available to model common errors ( unexpected error, unauthorized, etc )
Type Safe The Result supports generics to make the value it holds type safe
HTTP Status Easy to model Http status codes and also easy to convert a Result to an Http Response
API Design Improves API design by facilitating meaningful results/failures to originate from a service layer rather than from a Controller/API layer
CLI The general purpose design of the Result allows it to be used as the response for both HTTP and CLI(Command Line Interface) requests in Slate.

Source

package slatekit.common.common
jar slatekit-common.jar
source Result.kt and helpers to build HTTP compatible results ResultFuncs.kt

Example

  // Build a result using the Success "branch" of Result
  val result1 = Success(
      data = "userId:1234567890",
      code = SUCCESS,
      msg = "user created",
      tag = "tag001",
      ref = "XY123"
  )

  // Build a result using the Failure "branch" of Result
  val result2 = Failure<String>(
      code = BAD_REQUEST,
      msg = "user id not supplied",
      tag = "tag001",
      err = IllegalArgumentException("user id"),
      ref = null
  )
  
  // CASE 1: Success ( 200 )
  val res1 = success(123456, msg = "user created", tag = "promoCode:ny001")
  printResult(res1)  
  
  // CASE 2: Failure ( 400 ) with message and ref tag
  val res2a = failure<String>(msg = "invalid email", tag = "23SKASDF23")
  printResult(res2a) 
  
  // CASE 3: Unauthorized ( 401 )
  val res3 = unAuthorized<String>(msg = "invalid email")
  printResult(res3)
